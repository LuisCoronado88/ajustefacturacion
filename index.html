AjustePredictor

Sustentación de Arquitectura de Machine Learning para Modelado de Datos Desbalanceados

1. Contexto del Problema y Objetivos Estratégicos

En el sector de servicios públicos domiciliarios (agua y alcantarillado), las desviaciones físicas de consumo y las inconsistencias comerciales generan reclamaciones masivas por parte de los usuarios. Cuando estas disputas son procedentes, la compañía aplica un Ajuste de Facturación. Operar de manera reactiva ante este fenómeno afecta directamente el flujo de caja e inunda a los analistas de control de pérdidas con auditorías manuales.

Objetivos del Proyecto:

  • Detección Temprana: Identificar en la etapa de precrítica comercial qué contratos presentan alta probabilidad de requerir un ajuste el próximo mes.
  • Eficiencia Operativa: Filtrar quirúrgicamente el universo de usuarios para asignar inspecciones en terreno u órdenes de revisión solo a casos con anomalías reales comprobadas.

2. Análisis Exploratorio de Datos (EDA) e Ingeniería de Características

El modelo fue entrenado y validado utilizando un volumen histórico real de 659,856 registros comerciales.

🚨 Diagnóstico de Desbalanceo Extremo

La variable objetivo es VALOR_AJUSTE_PERIODO ($1$ = Requiere Ajuste; $0$ = Cuenta Estable). El EDA reveló la siguiente distribución:

  • Clase 0 (Sin Ajuste): 651,395 registros (98.72% del total).
  • Clase 1 (Con Ajuste): 8,461 registros (1.28% del total).

Al ser un problema de "aguja en un pajar", las métricas tradicionales de exactitud global inducirían a un error grave en la toma de decisiones.

Ingeniería de Características Senior (Feature Engineering):

Para dotar a los modelos de contexto de negocio, se estructuraron tres variables de contraste matemático a partir del diccionario comercial:

  • DESVIACION_M3: Representa la diferencia neta (M3_PERIODO - M3_PROMEDIO). Expone picos atípicos de consumo de agua de forma individualizada.
  • DESVIACION_FACTURACION: Captura variaciones abruptas de costos financieros (FACTURACION_PERIODO - FACTURACION_PROMEDIO).
  • ALERTA_ALTO_CONSUMO: Indicador binario que se activa si el consumo físico actual supera 1.5 veces el promedio de los últimos 6 meses.

3. Arquitectura de Métricas Avanzadas

El marco de evaluación descartó el Accuracy general. El éxito del negocio radica en optimizar tres métricas orientadas estrictamente a la **Clase 1 (Ajustes)**:

  • Precision: Mide la confianza de la alerta. Un porcentaje alto asegura que cuando el sistema marca una cuenta, el auditor no pierda tiempo en una falsa alarma.
  • Recall (Sensibilidad): Mide la capacidad de captura. Asegura cuántos ajustes verdaderos del pajar logramos rescatar.
  • F1-Score: Balance de ambas. Es la métrica reina utilizada para dictaminar el modelo campeón.

4. Modelos Entrenados y Validación Cruzada

Para asegurar la reproducibilidad y blindar el sistema contra el sobreajuste (overfitting), implementamos una Validación Cruzada Estratificada de 5 Pliegues (Stratified 5-Fold Cross-Validation). Se evaluaron tres familias de algoritmos alimentados mediante un pipeline robusto:

  1. Regresión Logística (Balanced): Configurada para ponderar fuertemente la clase minoritaria mediante penalizaciones de pesos.
  2. Random Forest Classifier (Balanced): Ensamble no lineal estructurado para penalizar desviaciones complejas.
  3. Gradient Boosting Classifier: Arquitectura secuencial de optimización por gradiente.

5. Selección del Mejor Modelo (Resultados Reales de Consola)

Tras el procesamiento multivariable de los 659,856 registros, la tabla de rendimiento consolidada arrojó las siguientes métricas exactas:

Modelo de Clasificación F1-Score (Métrica Selección) Recall (Sensibilidad) Precision (Confianza) Accuracy Global
Gradient Boosting 🏆 43.84% 29.19% 88.07% 99.04%
Random Forest (Balanced) 12.67% 58.50% 7.10% 89.66%
Regresión Logística (Balanced) 9.48% 61.93% 5.13% 84.84%

Veredicto Técnico y Argumentación Senior: El modelo ganador absoluto para producción es Gradient Boosting. Aunque los modelos balanceados incrementan el Recall, colapsan la operación destruyendo la precisión (caen a ~5%). En un entorno corporativo de servicios públicos, un modelo con una Precision del 88.07% es un éxito rotundo: significa que el algoritmo es quirúrgico. Genera pocas alertas, pero con la certeza casi absoluta de que el caso requiere intervención inmediata, optimizando los recursos humanos de control de pérdidas.

6. Empaquetamiento de la Solución

El software fue modularizado bajo principios de arquitectura limpia:

  • data/data_loader.py: Ingesta con parseo automático de comas decimales e ingeniería de variables.
  • train_models.py: Pipeline de validación cruzada multihilo automatizado.
  • app.py: Interfaz analítica e interactiva en Dash y Plotly que consume el modelo final para simulaciones en tiempo real por parte del usuario.
